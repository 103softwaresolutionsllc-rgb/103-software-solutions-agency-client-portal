# How 103 Agency OS Was Designed & Built

**Client:** 103 Software Solutions LLC  
**Stack:** React + Vite (frontend) · Express (backend) · PostgreSQL (database)  
**Author:** AI-assisted full-stack build on Replit

---

## 1. Starting Point

The project began from a ZIP file spec — a Next.js concept for a "Business OS" covering clients, projects, tasks, billing, phases, milestones, files, and activity logs. Rather than cloning the spec framework, I rebuilt it as a clean monorepo with a dedicated frontend and API server for better separation of concerns.

---

## 2. Architecture

The app lives in a **pnpm monorepo** with three packages:

```
/artifacts/agency-os        → React + Vite frontend (served at /)
/artifacts/api-server       → Express REST API (served at /api)
/lib/db                     → Drizzle ORM schema + migrations
/lib/api-client-react       → Auto-generated React Query hooks (via Orval)
/lib/api-spec               → OpenAPI 3.0 specification
```

### Data Flow

```
Browser → React (Vite) → /api/* → Express → PostgreSQL
                ↑
         React Query hooks
         (auto-generated from OpenAPI)
```

## 3. Database Design

Built with **Drizzle ORM** on **PostgreSQL**. The schema is multi-tenant ready — every query is scoped by `organizationId`.

| Table         | Purpose                                      |
|---------------|----------------------------------------------|
| organizations | Top-level tenant (103 Software Solutions LLC)|
| users         | Staff members with roles (admin / member)    |
| clients       | Agency clients with contact details          |
| projects      | Client engagements with budget & status      |
| phases        | Ordered stages within a project              |
| milestones    | Key deliverable checkpoints per phase        |
| tasks         | Work items with priority, due date, assignee |
| invoices      | Billing records linked to client + project   |
| activity_logs | Immutable audit trail of all actions         |
| files         | File metadata attached to projects           |

All tables include `created_at`, `updated_at` timestamps. Foreign keys enforce referential integrity throughout.

## 4. API Design

A **30+ endpoint REST API** was spec'd first in OpenAPI 3.0 YAML, then implemented in Express:

```
POST   /api/auth/login          Sign in, returns HMAC token
GET    /api/auth/me             Current user info
POST   /api/auth/logout

GET    /api/clients             List all clients
POST   /api/clients             Create client
GET    /api/clients/:id
PUT    /api/clients/:id
DELETE /api/clients/:id

GET    /api/projects            List projects (with task counts)
POST   /api/projects
GET    /api/projects/:id        Full detail with phases, milestones, tasks
PUT    /api/projects/:id
DELETE /api/projects/:id

GET    /api/tasks
POST   /api/tasks
PUT    /api/tasks/:id
DELETE /api/tasks/:id

GET    /api/invoices
POST   /api/invoices
PUT    /api/invoices/:id

GET    /api/phases
POST   /api/phases
GET    /api/milestones
POST   /api/milestones

GET    /api/activity            Paginated audit log
GET    /api/dashboard/metrics   Revenue, active projects, overdue invoices
```

**Authentication** uses a custom HMAC token (no third-party JWT library). The token is stored in `localStorage` and injected into every API request via a custom fetch wrapper. Auth state is managed in a React Context to avoid timing issues with query caching.

**Activity logging** is automatic — every write operation (create/update/delete) appends a record to `activity_logs`, capturing the actor, action type, entity, and a human-readable description.

## 5. Frontend Architecture

Built with **React 18 + Vite + TypeScript**, styled with **Tailwind CSS v4**.

### Code Generation

The OpenAPI spec powers automatic generation of:
- **TypeScript types** for every request/response shape
- **React Query hooks** (`useGetClients`, `useCreateProject`, etc.) via **Orval**

This means zero hand-written API call code on the frontend — the spec is the source of truth.

### Pages

| Route        | Description                                      |
|--------------|--------------------------------------------------|
| `/login`     | Auth with pre-filled demo credentials            |
| `/dashboard` | KPI cards, revenue chart (Recharts), activity feed |
| `/clients`   | Searchable client table with status & project count |
| `/projects`  | Card grid with progress bars, budget, due dates  |
| `/tasks`     | Kanban board (To Do / In Progress / Review / Done)|
| `/invoices`  | Billing table with status badges (Draft/Sent/Paid/Overdue) |

### Auth Flow

```
Login form submit
  → POST /api/auth/login
    → Token + user object returned
      → Stored in localStorage + React AuthContext
        → AuthGuard reads context (not HTTP) → instant, no flicker
```

The `AuthGuard` component wraps all protected routes and reads from React state rather than making an HTTP round-trip — this eliminates the race condition where a cached 401 error would block navigation after login.

## 8. Key Technical Decisions

| Decision | Why |
|---|---|
| Separate frontend + API server | Clean separation; API can be consumed by mobile apps later |
| OpenAPI-first | Single source of truth; codegen eliminates type drift |
| Drizzle ORM | Type-safe SQL with zero magic, easy to read raw queries |
| React Query | Automatic caching, loading states, and refetch on demand |
| HMAC token auth | No external auth dependencies; simple and auditable |
| React AuthContext | Avoids React Query cache timing issues on login/logout |
| pnpm monorepo | Shared types and utilities across packages; fast installs |

## 9. What Can Be Extended

The architecture is intentionally modular. Natural next additions would be:

- **Role-based permissions** — the `role` column is already on users; just add middleware guards
- **File uploads** — the `files` table schema exists; add S3/object storage integration
- **Real-time activity** — swap the polling activity feed for WebSocket push
- **Team management** — invite users to an organization
- **Client portal** — a read-only view for clients to track their own projects
- **Mobile app** — the Express API is already REST, ready for an Expo client
